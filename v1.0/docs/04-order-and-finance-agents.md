# 04 - Order & Finance Agents

This document covers the advanced decision-making and financial gating segments of the fully autonomous pipeline.

## The Order Agent (`order-agent`)

The Order Agent represents the "Brain" of the purchasing flow. It takes the scattered data generated by the Inventory Agent's requests and distills it into a concrete action.

### LLM-Driven Decision Making
The Order Agent receives a payload containing the current needs (e.g., "Restock 50 Webcams immediately") and a list of quotes from multiple suppliers (each with different prices, delivery days, and historical reliability scores).

It formulates a structured prompt and sends it to the **Google Gemini** model. The prompt instructs the LLM to act as a seasoned Procurement Manager.

The LLM is expected to analyze the trade-offs:
*   "Quote A is $1,200 and arrives in 2 days."
*   "Quote C is $800 and arrives in 7 days."
*   "Our inventory is critically low (0 units). We cannot afford a 7-day lead time despite the cost savings. We must prioritize delivery speed." => **Select Quote A.**

Conversely, if inventory is merely approaching the threshold (e.g., 14 units remaining), the LLM might reason that the 7-day wait is acceptable to save $400. => **Select Quote C.**

### Execution & Handoff
Once the LLM decides, the Order Agent constructs a formal Order Request object. Before committing this to the database, it must request capital. It sends the Order Request to the `finance-agent`.

## The Finance Agent (`finance-agent`)

The Finance Agent acts as the final gatekeeper preventing runaway AI spending. It strictly enforces budget constraints using simple, deterministic rules rather than relying on LLM hallucination for critical financial boundaries.

### Budget Management
The system operates on a predefined `MONTHLY_BUDGET` (e.g., $50,000), tracked in the PostgreSQL `budget` table.

When the Finance Agent receives a request to approve an order for $X:
1.  **Fund Check:** It checks the `budget` table. If `spent + X > total_budget`, the order is **Rejected**.
2.  **Auto-Approval:** If funds are available, it checks the `BUDGET_AUTO_APPROVE_PCT` (e.g., 30%). If $X < (30% of $50,000), the order is **Auto-Approved**.
3.  **Escalation:** If the order is very large (e.g., $20,000), the Finance Agent marks the transaction as **Pending/Escalated**. This simulates the need for human administrator override for massive purchases.

### Transaction Logging
Regardless of the outcome, every interaction is securely logged in the `budget_transactions` table to maintain a perfect audit trail.

If approved, the Finance Agent signals the Order Agent to proceed, the order is officially written to the `orders` table, and the `notification-agent` is alerted to broadcast the success.
